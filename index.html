Web Socket

La tecnologia WebSocket nasce per far sì che il web possa comunicare con i server in modo dinamico, efficiente e superare i limiti presenti nel protocollo HTTP o dal polling periodico dei dati che rendeva le pagine web limitate nella loro responsività e nella loro dinamicità.

Il WebSocket apre una connessione client-server persistente e bidirezionale (Full-Duplex), che permette ad entrambe le parti di inviare dati in qualsiasi momento. Il processo di apertura della connessione tra server e client avviene con il cosiddetto "handshake" (stretta di mano), ossia una chiamata al costruttore JavaScript:

Richiesta del client: 
const socket = new WebSocket("ws://server.example.com/mychat");
GET /mychat HTTP/1.1 
Host: server.example.com 
Upgrade: websocket 
Connection: Upgrade 
Sec-WebSocket-Key: x3JJHMbDL1EzLkh9GBhXDw== 
Sec-WebSocket-Protocol: chat 
Sec-WebSocket-Version: 13 
Origin: http://example.com
La risposta del server: 
HTTP/1.1 101 Switching Protocols 
Upgrade: websocket 
Connection: Upgrade 
Sec-WebSocket-Accept: HSmrc0sMlYUkAGmm5OPpG2HaGWk= 
Sec-WebSocket-Protocol: chat

La richiesta è strutturata in modo che il server possa gestirla inizialmente como una normale richiesta HTTP. La differenziazione delle chiamate deriva da alcuni campi specifici (header) che identificano la richiesta come WebSocket. Tecnicamente la chiamata funziona tramite il client che invia una chiave casuale nel campo Sec-WebSocket-Key. Questo valore viene ricevuto dal server e concatenato con un codice identificativo univoco standard (GUID). Il risultato viene passato alla funzione di hash SHA-1 e il suo digest viene codificato tramite Base64. La stringa ottenuta viene alla fine inserita nell’header della risposta Sec-WebSocket-Accept.

Il Ciclo di Vita

Una volta creata la connessione tra client e server il ciclo di vita si basa sui quattro eventi seguenti:

socket.onopen
socket.onmessage
socket.onerror
socket.onclose

socket.onopen

L’evento socket.onopen decreta l’avvenuta connessione tramite WebSocket tra il client e il server. Questo evento scatta nel momento in cui la connessione TCP avviene correttamente tra le parti. Ogni tentativo di invio dati tra client e server precedente all’attivazione dell’evento onopen tramite il metodo .send() va in errore poiché il canale non è ancora aperto. Alcuni esempi in cui scatta questo metodo può essere, tra gli altri casi, quello in cui viene inviata la richiesta di autenticazione al server da parte di un client. 

socket.onopen = (event) => { 
    socket.send(JSON.stringify({ action: "join", room: "sim-racing" })); 
};

socket.onmessage

L’evento socket.onmessage è il cuore della comunicazione in quanto gestisce lo scambio dei messaggi e dei dati tra client e server. Ogni volta che il server invia un pacchetto (called Frame) verso il client l’evento scatta lavorando in maniera asincrona, ossia non è necessario che il client faccia una richiesta preventiva. I dati arrivano all'interno della proprietà event.data e possono essere stringhe semplice (molto spesso JSON serializzati) o dati binari (Blob o ArrayBuffer). Un esempio è la ricezione di telemetria in tempo reale. 

socket.onmessage = (event) => { 
    const dati = JSON.parse(event.data); 
    console.log("Dati ricevuti:", dati); 
};

socket.onerror

L’evento socket.onerror scatta nel momento in cui si verifica un errore imprevisto nella comunicazione o nel protocollo sottostante. Come da standard W3C non viene mai specificato un dettaglio approfondito sul codice JavaScript riguardo al motivo dell’errore nella connessione per motivi di sicurezza. Un caso d'uso dell'interfaccia utente può essere quello in cui un messaggio non venga inviato e viene mostrato l'errore "Impossible connettersi al server". Questo evento viene quasi sempre utilizzato in coppia con l'evento di chiusura subito dopo l’errore. 

socket.onerror = (error) => { 
    console.error("Errore WebSocket rilevato:", error); 
};

socket.onclose

L’evento socket.onclose scatta nel momento in cui la connessione viene chiusa definitivamenre (in modo intenzionale via socket.close() o accidentale per crash/timeout). Questo evento, a differenza di onerror, possiede una numerosa quantità di utili informazioni riguardo al motivo della chiusura: possiede un event.code (codice numerico standard, es: 1000 per chiusura normale o 1006 per anomala) e un event.reason. È fondamentale per intercettare le disconnessioni e attivare la riconnessione manuale. 

socket.onclose = (event) => { 
    console.log(`Chiuso. Codice: ${event.code}, Motivo: ${event.reason}`); 
};

Socket.io Architecture

Analisi e ottimizzazione dei flussi di rete persistenti

Libreria di Astrazione Real-Time

Nonostante l'API WebSocket nativa sia uno strumento potente, lo sviluppo di applicazioni web in produzione si scontra spesso con i limiti fisici delle reti reali. Se un utente perde momentaneamente la connessione o naviga dietro un firewall aziendale rigido, il protocollo nativo interrompe la comunicazione senza fornire sistemi di recupero automatici.

Socket.io nasce come una **libreria di astrazione** progettata per risolvere queste criticità, garantendo una comunicazione bidirezionale persistente, robusta e ad alta affidabilità tra client e server.

Vantaggi di implementazione accademici:
• Rete resiliente con gestione automatica degli errori di timeout.
• Smistamento dati (Namespaces e Rooms) nativo integrato.

I Tre Pilastri Fondamentali

1. Riconnessione Automatica

Se la connessione si interrompe, la libreria avvia una logica in background usando un algoritmo di ritardo esponenziale per ripristinare il flusso senza interventi manuali.

2. Fallback HTTP Long-Polling

Se un ambiente blocca i WebSocket, Socket.io avvia la connessione tramite tradizionali richieste HTTP Long-Polling per garantire la totale stabilità di rete.

3. Namespaces e Stanze (Rooms)

Permettono al server di suddividere i client connessi e trasmettere pacchetti esclusivamente a gruppi selezionati, risparmiando banda.

Esempio implementazione client-side: 
const socket = io("https://server.example.com");
socket.on("connect", () => { console.log("Connesso:", socket.id); });
socket.on("telemetry_update", (data) => { console.log("Dati:", data); });
